VOI VIBECODE CHALLENGE
Event Demand Dashboard

Build a web app that gives Voi an overview of upcoming events in a city of your choice to aid in fleet allocation and customer acquisition.
The challenge
London, Paris, and Berlin are three of Voi's biggest markets, and each city hosts hundreds of large events per year — concerts, football matches, festivals, marathons — each generating predictable surges in micromobility demand. A tool that surfaces these events, maps them against Voi infrastructure, and flags where to pre-position vehicles would turn a reactive ops team into a proactive one. Think you can build it in 150 minutes?
  • —Ops teams currently rely on historical ride data and intuition to anticipate demand — there is no structured view of what is happening in the city on a given day.
  • —Vehicles are not always where they need to be when large events end — leading to missed rides, poor customer experience, and lost revenue.
  • —Events are a strong customer acquisition opportunity — first-time riders often try Voi when they see vehicles near a venue — but this is currently left to chance.
  • —Planning is fragmented across calendars, spreadsheets, and manual checks — with no single source of truth for upcoming high-demand moments.
Potential approaches

Think about how an ops manager starts their morning. What do they need to see first? How do they move from overview to action? Your prototype should answer these questions in whatever form makes most sense to you — a calendar, a map, a ranked list, a single event deep-dive, or something else entirely.

Surfacing upcoming high-demand events at a glance — and making it easy to filter by date, type, or demand level.

Even a simple demand flag — high / medium / low — goes a long way toward making the output actionable rather than just informational.

Thinking spatially adds depth: which events fall outside the Voi zone? Where is parking thin relative to expected demand?

The Voi zone and parking data are there to be used — even a light touch of geographic context changes how useful the tool feels.

Data sources
External PredictHQ event data — upcoming events with attendance, coordinates, and demand rank, pre-fetched for London, Berlin, and Paris. Pick one city or go multi-city if you're feeling ambitious. Files are available in the data folder below.
Internal Voi map data — operations zone and designated parking spots for your chosen city. Required context for any fleet or zone analysis.

What we're looking for
  • 1Clear overview — someone unfamiliar with the data should open it and immediately understand the event landscape in the city.
  • 2Actionable — it should be obvious what an ops team should do next, without needing to dig or interpret raw data.
  • 3Intuitive — no explanation needed to navigate it. Visual hierarchy and design should carry the user through.
  • 4Grounded in data — outputs and recommendations should be traceable back to the provided dataset, not arbitrary.
  • 5Scalable thinking — does the approach generalise across cities? A solution that works for London with a data swap for Berlin is more valuable than one hardcoded to a single place.
Tips & tricks
  • —Voi's approximate fleet sizes: London ~5,500 vehicles, Paris ~6,000, Berlin ~12,000 — use these as references when sizing fleet recommendations.
  • —The provided dataset contains more fields than you need. Be selective — focus on the data points that are material to fleet decisions and discard the rest. A clean, focused data model will serve you better than trying to surface everything.
  • —There will be a lot of information on screen — use visual hierarchy and colour deliberately. The tool should make it immediately obvious which events need attention and what action to take.

Data documentation ▶
Voi map data

Two GeoJSON files are provided per city — an operations zone and a parking spots file. Both follow the standard FeatureCollection structure and can be loaded directly into any map library.

  • —operations.json — a single MultiPolygon feature defining the entire Voi operations zone for the city. This is the boundary within which vehicles may be ridden and parked. Use this to check whether an event falls inside or outside the zone, and to render the service area on the map.
  • —parking.json — individual Polygon features, each representing a designated parking spot within the ops zone.

Voi uses mandatory parking — riders must end their trip at or near a designated parking spot inside the ops zone. Parking spot density around an event venue is therefore a direct proxy for how well Voi can serve that location. An event inside the zone with few nearby parking spots is a signal to pre-position vehicles proactively.

PredictHQ event data

PredictHQ is a demand intelligence platform that aggregates and structures real-world event data from thousands of sources globally. Event data for London, Berlin, and Paris has been pre-fetched and is provided in the data folder. Each file is a JSONL (JSON Lines) file — one JSON object per line, each representing a single event. If you want to explore PredictHQ directly or fetch additional data, sign up at predicthq.com — the next 90 days of event data are free.

These are tips to get you started — not a complete guide. The best way to understand what's in this dataset is to query it with AI. Ask questions, explore the fields, and iterate. You'll find what's useful faster that way than reading documentation.

Each record has many fields — most of which you won't need. The fields most relevant to this challenge are:

title category start_local end_local location phq_attendance rank local_rank phq_labels geo duration scope description entities state
  • —phq_attendance is the predicted attendance figure — your primary demand signal. It will be null for some records (public holidays, observances), so always handle the null case and fall back to rank or local_rank as a proxy. Some values may be outliers — PredictHQ aggregates from multiple sources including scraped data, so sanity-check any figures that look unexpectedly large.
  • —location is a [longitude, latitude] array — this follows the GeoJSON spec, which defines coordinate order as lon/lat. This is intentional but counterintuitive, and the opposite of what most map libraries (e.g. Leaflet) expect. Swap the values when passing to your map. geo contains a full GeoJSON polygon for events with a defined area, useful for drawing event zones on the map.
  • —category groups events by type. The most useful for fleet planning are typically concerts, performing arts, and sports. Categories like public holidays and observances are low-signal for this use case and can likely be filtered out.
  • —rank (0–100) is PredictHQ's global impact score; local_rank is the same score relative to the local area. For city-level recommendations, local_rank is generally the more useful signal.
